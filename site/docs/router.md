# File-Based Router

Auwla ships a client-side router built directly on the runtime's reactivity. URL changes invalidate only the components that read route state â€” no virtual DOM, no manual `commit()`, and no stored component handles.

The `auwlaRouter()` Vite plugin turns a `src/pages/` directory into a fully typed, code-split route table. If you prefer to write routes by hand, see the [Router API](./router-api) guide.

---

## Setup

```ts
// vite.config.ts
import { defineConfig } from 'vite';
import { auwla } from 'auwla/vite';
import { auwlaRouter } from 'auwla/vite-router';

export default defineConfig({
  plugins: [
    auwla(),
    auwlaRouter({ dir: 'src/pages' }),
  ],
});
```

```tsx
// src/App.tsx
import { Router } from 'auwla/router';
import routes from 'auwla:routes'; // virtual module generated by the plugin

export default function App() {
  return () => <Router routes={routes} suspend />;
}
```

The `auwla:routes` import is explicit â€” the plugin does not magically rewire the `<Router>` component.

## File Conventions

```
src/pages/
  index.tsx          â†’ /
  about.tsx          â†’ /about
  _layout.tsx        â†’ wraps sibling pages
  posts/
    index.tsx        â†’ /posts
    [id].tsx         â†’ /posts/:id
    [id]/
      comments.tsx   â†’ /posts/:id/comments
  [...404].tsx       â†’ /*  (catch-all)
```

Rules:

- `index.tsx` at any level maps to that directory's root path.
- `[param]` becomes `:param` (e.g., `[id]` â†’ `/posts/:id`).
- `[...name]` becomes `*` (catch-all wildcard).
- Nesting directories adds path segments.

Supported extensions: `.tsx`, `.ts`, `.jsx`, `.js`. Configurable via the `extensions` option.

## Named Exports

Page files can export route fields. The plugin detects which ones exist and wires them automatically.

| Export | Route field |
| :--- | :--- |
| `default` | `component` |
| `routed` | `routed` (async loader) |
| `pending` | `pendingComponent` |
| `error` | `errorComponent` |
| `guard` | `guard` |
| `meta` | `meta` |

```tsx
// src/pages/posts/[id].tsx
import { getRouted, type RouteContext } from 'auwla/router';

export const routed = async (ctx: RouteContext<'/posts/:id'>, signal: AbortSignal) => {
  const res = await fetch(`/api/posts/${ctx.params.id}`, { signal });
  return res.json();
};

export default function PostDetail() {
  const data = getRouted(routed);

  if (data?.pending)  return <p>Loadingâ€¦</p>;
  if (data?.rejected) return <p>Error: {String(data.reason)}</p>;

  return () => <h1>{data?.value?.title}</h1>;
}
```

## Layouts

A `_layout.tsx` file wraps all pages in its directory and subdirectories.

```tsx
// src/pages/_layout.tsx
import type { RouteComponent } from 'auwla/router';

export default function AppShell(Child: RouteComponent) {
  return () => (
    <div class="app">
      <nav>â€¦</nav>
      <main><Child /></main>
    </div>
  );
}
```

The layout receives the raw route component function, not an element. Calling `<Child />` creates the element and lets the runtime manage its lifecycle.

### Nested layouts

```
src/pages/
  _layout.tsx           â†’ root layout
  index.tsx
  dashboard/
    _layout.tsx         â†’ dashboard layout
    index.tsx           â†’ /dashboard
    users.tsx           â†’ /dashboard/users
```

For `/dashboard/users`, the generated component is roughly `AppShell(() => DashShell(DashboardUsers))`. The outermost layout renders first, and `DashShell` is nested inside `AppShell`'s `<Child />` slot.

### Layout guards

Export a `guard` function from `_layout.tsx` to protect all routes in that directory:

```tsx
// src/pages/dashboard/_layout.tsx
import type { RouteContext, RouteComponent } from 'auwla/router';

export const guard = (ctx: RouteContext) => {
  if (!auth.isDashboardUser()) return '/login';
};

export default function DashShell(Child: RouteComponent) {
  return () => <div class="dashboard"><Child /></div>;
}
```

Guard execution order for a page inside nested layouts runs from most specific (page) to least specific (outer layout).

## Route Loaders (`routed`)

`routed` is an async function that loads data before (or while) a route renders. The router automatically cancels the loader if the user navigates away.

```tsx
export const routed = async (ctx: RouteContext<'/posts/:id'>, signal: AbortSignal) => {
  const res = await fetch(`/api/posts/${ctx.params.id}`, { signal });
  return res.json();
};

export default function PostDetail() {
  const data = getRouted(routed);

  return () => (
    <article>
      {data?.pending && <p>Loadingâ€¦</p>}
      {data?.rejected && <p>Error: {String(data.reason)}</p>}
      {data?.resolved && <h1>{data.value.title}</h1>}
    </article>
  );
}
```

The handle returned by `getRouted()` exposes `.pending`, `.resolved`, `.rejected`, `.value`, `.reason`, `.status`, `.cancel()`, and `.refresh()`.

## Error & Pending UI

You can handle errors inline (as shown above) or via route-level exports.

### Route-level error component

```tsx
// src/pages/posts/[id].tsx
export function error() {
  const err = getRouteError();

  return () => (
    <div>
      <h2>Could not load post</h2>
      <p>{String(err?.reason)}</p>
      <button onClick={() => err?.retry()}>Try again</button>
    </div>
  );
}
```

When an `error` export exists, the router renders it instead of the default component if the loader rejects. `getRouteError()` only works inside error components.

### Pending component

```tsx
export const pending = () => <p>Loading postâ€¦</p>;
```

`pending` is shown while `routed` is in flight. It is ignored during suspend navigations â€” the previous route stays visible instead.

## Reading Route State

Call these helpers in component setup. They read from the router's current context.

```tsx
import { getParams, getQuery, getLocation, getRouteMeta } from 'auwla/router';

function PostDetail() {
  const { id } = getParams('/posts/:id');  // { id: string }
  const query = getQuery();                 // ?tab=comments â†’ { tab: 'comments' }
  const loc = getLocation();                // '/posts/42?tab=comments'
  const meta = getRouteMeta<{ title: string }>();

  return () => <h1>Post {id}</h1>;
}
```

## Navigation

### `<Link>`

```tsx
<Link href="/posts">Posts</Link>
<Link href="/posts/:id" params={{ id: '3' }}>View Post</Link>
<Link href="/posts" replace>Replace history</Link>
<Link href="/posts" activeClass="active">Nav item</Link>
<Link href="/posts" exactActiveClass="active">Exact match</Link>
```

### Programmatic navigation

```tsx
import { navigate, back, forward } from 'auwla/router';

navigate('/posts');
navigate('/posts/:id', { id: '3' });
navigate('/login', { replace: true });
back();
forward();
```

### Active route checks

```tsx
import { isActive, isExactActive } from 'auwla/router';

isActive('/posts');      // true for /posts and /posts/3
isExactActive('/posts'); // true only for /posts
```

## Suspend Mode

```tsx
<Router routes={routes} suspend />
```

When suspend is enabled and a route has a `routed` function:

- The old route stays visible while the new route loads.
- The `<html>` element gets a configurable CSS class/attribute (default: `html.suspended`).
- The new route commits only after the loader resolves.
- On initial load, the route's `pending` export is shown.

```tsx
<Router
  routes={routes}
  suspend={{
    class: 'is-loading',
    attr: 'data-loading',
  }}
/>
```

## Typed Navigation

The plugin writes `src/auwla.gen.ts`, which augments the `Register` interface. Once that file is imported anywhere in your project, `navigate()`, `<Link href>`, `isActive()`, etc. validate against your real routes.

```ts
navigate('/posts/:id', { id: '3' }); // âś…
navigate('/posts/:id');              // âťŚ missing params
navigate('/nope');                   // âťŚ unknown route
```

Commit `src/auwla.gen.ts` to version control so CI and editor type-checking work without running the dev server. The plugin regenerates it automatically on startup.

## Plugin Options

```ts
auwlaRouter({
  dir: 'src/pages',            // pages directory
  genFile: 'src/auwla.gen.ts', // type output
  lazy: false,                 // code-split routes with loaders
  extensions: ['.tsx', '.ts'], // page file extensions
});
```

| Option | Default | Description |
| :--- | :--- | :--- |
| `dir` | `'src/pages'` | Pages directory, relative to Vite root. |
| `genFile` | `'src/auwla.gen.ts'` | Where to write the typed route registration file. |
| `lazy` | `false` | Code-split every route that has a `routed` function into its own chunk. |
| `extensions` | `['.tsx','.ts','.jsx','.js']` | File extensions to treat as pages. |

When `lazy: true`, the `<Link>` component prefetches route chunks on `mouseenter` / `touchstart` for instant-feeling navigations.

---

## HMR

During development, the plugin watches the pages directory. When a page or layout file is added, removed, or renamed, the plugin invalidates the virtual module, rewrites `auwla.gen.ts`, and triggers a full page reload. Changes to the content of a page component are handled by Vite's normal HMR and do not trigger a full reload.

---

## When to Use the Low-Level API

File-based routing covers most apps. If you need dynamic routes, programmatic route tables, or runtime route composition, use the [Router API](/docs/router-api) (`defineRoutes`, `group`, `Outlet`, etc.) instead.
