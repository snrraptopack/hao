### Part 1: Security Audit Findings

**1. Cross-Site Request Forgery (CSRF) via `multipart/form-data`**
*   **The Threat:** The RPC endpoint currently accepts `multipart/form-data` to support file uploads and form submissions. The problem is that browsers consider this a "Simple Request." Unlike JSON submissions, the browser does *not* send a CORS preflight (`OPTIONS`) request before sending a multipart form.
*   **The Exploit:** An attacker can host a hidden `<form>` on their malicious website pointing to your `/_auwla/rpc` endpoint. If a user visits that site while logged into your app, their browser will automatically attach their session cookies and execute the Auwla mutation (e.g., deleting their account).
*   **The Mitigation Strategy:** Because Auwla is a Single Page Application (SPA), the most robust defense is to require a custom HTTP header (like `x-auwla-rpc: 1`) on all requests from the client. Standard HTML forms cannot send custom headers, and attempting to send them via `fetch` from a malicious domain will trigger a CORS preflight block.

**2. Prototype Pollution & Denial of Service**
*   **The Threat:** The RPC adapter resolves which server function to run by looking up the incoming key directly on the manifest object (`manifest[payload.key]`).
*   **The Exploit:** If an attacker sends `{"key": "__proto__"}`, they access the prototype chain of the manifest object. The adapter will attempt to read routing properties off the prototype, fail, and pass `undefined` downstream. If spammed, this unexpected type failure can cause unhandled exceptions that crash the Node/Bun process (Denial of Service).
*   **The Mitigation Strategy:** Ensure that manifest lookups are strictly checked using `Object.hasOwn()` rather than direct indexing.

**3. Error Information Leakage**
*   **The Threat:** The `errorResponse` function serializes all caught errors and sends `error.message` to the client.
*   **The Exploit:** While great for development, in production, if a database query fails or a connection string is misconfigured, the underlying driver's raw error message (which often contains internal IPs, schema names, or secrets) will be leaked directly to the browser network tab.
*   **The Mitigation Strategy:** Implement an error sanitizer that only passes through `error.message` for known, safe framework errors (like `ValidationError`). For all other generic errors, obscure the output to "Internal Server Error" in production.

**4. Middleware Double Execution Footgun**
*   **The Threat:** In the current `runMiddleware` pipeline, the `next()` function is passed down the chain.
*   **The Exploit:** If a developer makes a mistake in their custom middleware and awaits `next()` twice, the downstream handler (the remote function) will execute twice in the same request lifecycle. This can cause severe state corruption or double-billing in production.
*   **The Mitigation Strategy:** Implement a lock inside the pipeline execution loop that throws a strict error if `next()` is called more than once per middleware instance.

### Part 2: Seamless Native Integration (The "Platform" Concept)

Since Auwla relies on existing server frameworks (Bun, Hono, Express) to handle the actual HTTP layer, reinventing security middleware (CSRF, Rate Limiting, Session Management) inside Auwla is unnecessary and error-prone.

Here is the conceptual strategy for seamlessly integrating Auwla with the native ecosystem:

**The Routing Limitation**
Because Auwla funnels every remote function through a single `/_auwla/rpc` route, the host framework (e.g., Hono) has no idea if the request is trying to `createPost` or `deleteUser`. Therefore, you still need Auwla's `defineMiddleware` for *function-specific* logic (like validating the specific payload for a specific function).

**The Solution: Platform Context Injection**
For *global* logic (Auth, CSRF, Rate Limiting), you should rely entirely on the native framework. We can achieve this by expanding Auwla's `ServerContext` to include an agnostic `platform` object.

1.  **Host Middleware First:** A developer mounts their native Hono or Express middleware globally. This handles the heavy lifting—validating CSRF tokens, parsing JWTs, or pulling session data from Redis—and stores it in the native framework's context object (e.g., Hono's `c` object).
2.  **Adapter Injection:** The Auwla adapter wrapper (e.g., `createHonoAdapter`) captures that native context object and passes it into the WinterCG `fetch.ts` adapter under the `platform` key.
3.  **Native Access in Auwla:** Inside any `.server.ts` remote function, the developer has direct access to `ctx.platform`. They can call `ctx.platform.hono.get('user')` or use native response helpers.

**Why this is the ideal architecture:**
It keeps Auwla incredibly lightweight and WinterCG-compliant at its core, while allowing developers to leverage the massive, battle-tested plugin ecosystems of Hono, Express, or Bun for complex server infrastructure.
